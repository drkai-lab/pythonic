Навык для Grok, который заставляет его писать Python так, как написал бы настоящий senior-разработчик, а не обычный модельный мусор.

## Установка

**Для личного использования** (подгружается во всех разговорах):

```powershell
git clone https://github.com/drkai-lab/pythonic.git "$env:USERPROFILE\.grok\skills\pythonic"
```

**Внутри проекта** (для конкретного репозитория или команды):

```powershell
# запускай из корня своего репозитория
git clone https://github.com/drkai-lab/pythonic.git .\.grok\skills\pythonic
```

Никаких специальных инструментов не требуется. Обычный git clone. После добавления просто перезагрузи навыки в Grok.

Для Claude Code, Cursor, Windsurf и прочих инструментов: `skill.md` — это просто текст. Можно вставить таблицу антипаттернов в правила проекта, конвертировать в .mdc или скормить модели целиком. Ценность именно в конкретных заменах.

## Что реально меняется

Современные аннотации типов вместо импортов из 2018 года:

```python
# то, что большинство моделей всё ещё выдают
from typing import List, Optional, Dict
def process(items: List[str]) -> Optional[Dict[str, int]]: ...

# то, что выдаёт этот навык
def process(items: list[str]) -> dict[str, int] | None: ...
```

Обработка путей и итерации, от которых не хочется морщиться:

```python
# before
import os
for f in os.listdir("data"):
    if f.endswith(".txt"):
        ...

# after
from pathlib import Path
for p in Path("data").glob("*.txt"):
    ...
```

Полное содержание (структурное сопоставление с образцом против цепочек if-elif, когда стоит использовать `dataclass(frozen=True, slots=True)`, `asyncio.TaskGroup`, `zoneinfo`, таблица из 25+ антипаттернов, матрица фич по версиям и десять жёстких правил) полностью в [skill.md](skill.md).

## Что на самом деле делает этот навык

Целится в Python 3.10+, по умолчанию использует идиомы 3.12+ где они уже есть. Все публичные функции и имена на уровне модуля обязаны иметь настоящие аннотации типов. Где стандартная библиотека уже может — используем её, а не тащим сторонние пакеты. Тесты в стиле pytest, а не unittest.TestCase. Никаких голых `except:`, никаких изменяемых дефолтов, никаких `# type: ignore` без внятного обоснования.

Цель простая: перестать выкатывать код, который выглядит так, будто LLM выучился на старых ответах со Stack Overflow.

Если ты работающий Python-разработчик и когда-нибудь вздыхал, видя, как ИИ выдаёт тебе `os.path.join` и `typing.List` — этот навык именно для исправления такой ерунды.

Если в реальных кодовых базах уже прижились новые привычки — пиши issue или присылай PR. Остальное уже расписано в skill.md.
